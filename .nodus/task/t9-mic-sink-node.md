# t9 — Виртуальный микрофон на холсте — это ПРИЁМНИК (сток), не источник

**Статус:** 🟡 ui-bridge + визуал; движок уже готов
**Ждёт:** удобно делать в рамках редизайна UI
**Кто делает:** ui-bridge (логика подключения), разработчик (визуал ноды/порта)

## Суть простыми словами
Виртуальный микрофон имеет двойную природу: снаружи (для Discord) это микрофон,
из которого читают; внутри Nodus это **приёмник** — в него *вливают* звук
(`Spotify → Nodus Mic`), и этот звук Discord потом слышит. Сейчас на холсте он
ошибочно нарисован как источник с выходным портом (потому что Windows отдаёт его
как «устройство записи», и Library сделала из него source-ноду). Из-за этого
пользователь не может построить маршрут: «у источника выход и у микрофона выход».

## Критерий готовности
- [ ] Нода «Nodus Virtual Mic» на холсте — приёмник: входной порт, принимает
      входящие рёбра `источник → микрофон`
- [ ] Маршрут `Spotify → Nodus Virtual Mic` строится мышкой и работает
      (звук уходит в виртуальный микрофон)
- [ ] Обычные (реальные) микрофоны по-прежнему источники — не задеть их

## Технические заметки (для меня)
- Движок УЖЕ готов: `to_is_virtual_mic` в graph.rs определяет назначение по
  label, engine роутит такой маршрут в `VirtualRender` (t4, проверено в поле).
- Расходится только представление в UI: виртуальный микрофон (Windows
  DeviceType::Input + имя «nodus…mic») надо в холсте показывать как сток/выход,
  а не как source-ноду. Это правка моста (классификация ноды) + визуала (порт
  слева, стиль выходной ноды).
- Зона ответственности: семантика «сток» — логика (Claude/ui-bridge); позиция
  порта и вид ноды — визуал (разработчик). Делать вместе.
- Естественно вписывается в редизаайн (.nodus/docs/ui-redesign-prompt.md):
  там виртуальный микрофон — отдельный фиолетовый тип-назначение.
