# t8 — Имена устройств в Windows: «Динамики (Nodus)» / «Микрофон (Nodus Mic)»

**Статус:** 🟡 правка драйвера; механизм проверить в поле (вместе с t5 step 2b)
**Ждёт:** ничего (но удобно совместить с kernel-итерацией t5 step 2b)
**Кто делает:** driver

## Суть простыми словами
Сейчас в Windows оба наших устройства подписаны одинаково — «(Nodus Virtual
Audio)»: колонка как «Динамики (Nodus Virtual Audio)», микрофон как «Микрофон
(Nodus Virtual Audio)». Хочется короче и различимо: **«Динамики (Nodus)»** и
**«Микрофон (Nodus Mic)»**.

## Критерий готовности
- [ ] Колонка отображается как «Динамики (Nodus)»
- [ ] Микрофон отображается как «Микрофон (Nodus Mic)»
- [ ] Имена не сломали обнаружение в движке (детект по подстроке «nodus» цел)

## Технические заметки (для меня)
- Полевой факт 13.06: на тестовой Win10 endpoint-имя строится как
  `форм-фактор (DeviceDesc)`, а НЕ из FriendlyName пина (Wave.Desc
  «Nodus Virtual Speaker» был проигнорирован, победил DeviceDesc
  «Nodus Virtual Audio»). Поэтому простого пути «поменять Wave.Desc» мало.
- Варианты (проверить, какой реально переименовывает на этой Windows):
  (а) разные DeviceDesc — но устройство одно (один devnode), DeviceDesc общий;
  (б) запись PKEY_Device_FriendlyName в ключ endpoint'а (тот же механизм, что
      architect заложил для динамических устройств в ADR §6.2 — драйвер пишет
      FriendlyName в registry-ключ интерфейса из kernel);
  (в) проверить, не возьмёт ли Windows имя из топологии (KSNODETYPE имя пина).
- Детект в движке (`is_nodus_virtual_name` / `is_nodus_virtual_mic_name`) по
  подстроке «nodus» / «nodus»+«mic|микрофон» — новые имена ему подходят, но
  перепроверить тестом.
- Совмещать с t5 step 2b: в той kernel-итерации и так трогаем драйвер + INF.
