## 1. Что за проект

**Nodus** — десктопное приложение (Tauri) для визуального роутинга звука на Windows:
холст с нодами вместо табличного микшера. Пользователь соединяет источники звука
(Spotify, Discord, игра, микрофон) с выходами (наушники, OBS, виртуальный микрофон)
рёбрами графа; на каждом маршруте — своя громкость/mute/эффекты.

**Главное правило проекта — разделение труда (см. CLAUDE.md, оно ЖЁСТКОЕ):**
- **Человек (разработчик)** делает весь визуал UI: React-компоненты, CSS, layout, дизайн.
- **Claude** делает весь Rust-backend, kernel-драйвер и подключение логики к готовому
  визуалу (invoke, события, сторы) — **не меняя внешний вид ни на пиксель** без
  явного разрешения.

**Стиль общения с пользователем:** русский, простым языком, без жаргона.
Пользователь явно попросил НЕ использовать формат «Фаза 1/2/3» — вместо этого
говорить «что сделано → что это значит → какой следующий шаг». Задачи он читает
в `.nodus/task/backlog.md` (на русском, простым языком) — поддерживай его актуальным.

## 2. Состояние по слоям

### UI (зона человека) — визуально готов
Canvas, ноды, edges, инспектор, мульти-сцены — нарисовано. Claude к визуалу не прикасается.

### Rust backend (src-tauri/src/) — MVP в основном работает
- WASAPI enumerate устройств, реальный routing Source → Output, per-route
  mute/volume/pan (DSP в рендерере), splitter (tokio broadcast fanout), mixer.
- Автодетект процессов по .exe + real-time события; per-app изоляция через
  WASAPI process loopback.
- Интеграция с UI сделана: Library показывает реальные устройства/процессы,
  VU-метры, save/load сцен, solo.
- ~30 unit-тестов. Правила: никаких `.unwrap()` в проде, все Tauri-команды только
  в `commands/bridge.rs`, каждый модуль с тестами.
- `routing/engine.rs`: `Backend::Virtual` читает звук из кольца kernel-драйвера
  (`audio/virtual_capture.rs`), с fallback на WASAPI loopback, если драйвера нет —
  приложение обязано работать и без драйвера.

### Kernel-драйвер (src-tauri/driver/nodus_audio/) — ГЛАВНЫЙ ФРОНТ РАБОТ
Виртуальные аудиоустройства Windows умеет создавать только kernel-драйвер.
Это самая сложная и рискованная часть проекта. История и состояние — ниже.

## 3. Драйвер — история и текущее состояние

### История (важно, чтобы не повторять ошибки)
1. Первая версия — самописный WaveRT-драйвер с нуля. Итог: **2 подтверждённых BSOD**
   на реальной машине, и ревью 11.06 показало, что endpoint архитектурно не мог
   появиться в Windows (один wave-фильтр без topology-миниnoрта).
2. **Решение: списать и переписать на базе Microsoft SYSVAD** (официальный сэмпл
   виртуального аудиодрайвера, audio/sysvad из Windows-driver-samples, MIT).
   План миграции: `.nodus/sysvad_migration_plan.md`.
3. Каркас SYSVAD-версии написан (коммит `910e55a`): wave-миниnoрт (host-пин +
   bridge-пин) + topology-миниnoрт (bridge + connector `KSNODETYPE_SPEAKER`),
   связаны `PcRegisterPhysicalConnection`. Звук пока отбрасывается — каркас
   проверяет только «устройство видно, BSOD нет».
4. Ревью каркаса нашло повторённый баг старой версии: `CategoryCount = 0` в обоих
   `PCFILTER_DESCRIPTOR`. **Категории обязаны быть в дескрипторе** (PortCls по ним
   регистрирует device interfaces; INF AddInterface сам по себе их не активирует).
   Исправлено: wave = AUDIO/RENDER/REALTIME, topo = AUDIO/TOPOLOGY (коммит `6375393`).
5. INF-фикс: InfVerif error 1199 — DIRID 13 требует декорацию секции моделей
   `NTamd64.10.0...16299` (Win10 1709+) (коммит `324a19f`).
6. **CI впервые собрал SYSVAD-пакет: run 27311703010, success.** Артефакт
   `nodus_audio-driver-x64-Release`: .sys (test-signed), .inf, .cat, .cer, PDB,
   install.ps1/uninstall.ps1.

### Где код
- **`main`** — с 11.06.2026 рабочая ветка для ВСЕГО (решение пользователя:
  ветки создаём только при крайней необходимости). SYSVAD-драйвер влит
  (merge `8d35781`), старый самописный драйвер удалён при вливании.
- Ветка `feat/sysvad-driver` — историческая, больше не используется.

### Незыблемые kernel-правила (каждое — прошлый или предотвращённый BSOD)
- DPC/DISPATCH_LEVEL не трогает pageable и process-context память; section view
  для DPC — только `MmMapViewInSystemSpace` (или копирование в PASSIVE-потоке).
- Teardown: `KeCancelTimer` + **`KeFlushQueuedDpcs()`** до освобождения памяти,
  которую видит DPC.
- `ZwMapViewOfSection` ↔ `ZwUnmapViewOfSection` (НЕ `MmUnmapLockedPages`).
- stdunk `CUnknown` стартует с refcount 0: фабрика делает AddRef; канонический
  паттерн — `InstallSubdevice` в adapter.cpp.
- Именованные секции — с явным SDDL (read для обычных пользователей, иначе
  `OpenFileMappingW` из Nodus получит ACCESS_DENIED).
- Счётчики байт — 64-битные (ULONG при 384 кБ/с переполняется за ~3 часа).
- Никаких побайтовых копий в DPC — `RtlCopyMemory` максимум двумя сегментами.
- Имена секций параметрические сразу: `Global\NodusRing-<id>` (задел под t5).

### CI (`.github/workflows/driver.yml`)
- Триггер: push с изменениями в `src-tauri/driver/**` в ЛЮБУЮ ветку (починено:
  раньше tags-фильтр без branches глушил все push-триггеры) + workflow_dispatch.
- Сборка через EWDK ISO (~19 ГБ, качается каждый ран, ~20 мин). Тест-подпись.
- `gh` CLI на машине НЕТ. Статус ранов — через GitHub API (repo публичный:
  `voidless-labs/Nodus`); логи анонимно дают 403 — брать токен из
  `git credential fill`.

## 5. Команда агентов — `.claude/agents/`

Шесть субагентов (адаптация фреймворка из D:\claude\claude-agent под Nodus).
**Основная сессия = Team Lead**: говорит с человеком (по-русски), ставит задачи,
делегирует, принимает работу. Подробности — `.claude/agents/README.md`.

- `driver` — kernel C++/PortCls/INF/driver-CI. Kernel-правила зашиты в промпт.
- `backend` — Rust/WASAPI/Tauri (src-tauri/src/**).
- `ui-bridge` — подключение invoke/событий к готовому React UI. ЖЁСТКО: никаких
  визуальных изменений (JSX/CSS/layout) без явного разрешения человека в задаче.
- `qa` — верификация (критерии → проверки → доказательства) и ревью (вердикт
  APPROVE/REQUEST_CHANGES/BLOCK, шкала BLOCKER/CRITICAL/MAJOR/MINOR). Для драйвера —
  только статическое ревью + CI; рантайм честно помечается «не проверено».
- `architect` — дизайн/ADR в `.nodus/docs/` до кода (например, IOCTL-протокол t5).
- `docs` — документация, актуальность доски и статусов Roadmap в CLAUDE.md.

Правила оркестрации: максимум 3 агента параллельно (CLAUDE.md, правило №7), зоны
изменений не пересекаются (два агента никогда не пишут один файл), нетривиальные
изменения проходят qa-ревью, отчёты агентов — в `.nodus/report/r{N}-<slug>-<DDMMYYYY>.md`
с русской выжимкой первой строкой. Специалисты не редактируют backlog.md.

## 6. Прочие факты и правила работы

- ОС: Windows 11, shell PowerShell 7. Репо: `D:\claude\Nodus`,
  remote `https://github.com/voidless-labs/Nodus.git`.
- `.nodus/` КОММИТИТСЯ в репо (это не gitignored-workspace).
- Тестовая машина для драйвера — расходный Win10-ноут пользователя, Test Mode
  включён. PDB в артефакте — для символизации BSOD (`nodus_audio.sys+offset`).
- Подпись: тест-подпись только для разработки; для пользователей нужен attestation
  signing через Microsoft Partner Center (EV-сертификат, недели лид-тайма) — t6.
- Коммиты/пуши — только по просьбе пользователя.
- Автономность: при неоднозначности — принять решение, зафиксировать его
  (в отчёте/задаче) и двигаться; при блокере — пометить задачу и взять следующую
  независимую. Состояние фиксировать в файлах ДО конца сессии, а не в конце:
  обрыв контекста не должен терять прогресс.
- У Claude есть постоянная память проекта (memory) — она дополняет этот файл,
  но этот файл — самодостаточный минимум.
